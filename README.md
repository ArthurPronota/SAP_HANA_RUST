## SAP HANA — описание базы данных

**SAP HANA** — высокопроизводительная **in-memory** СУБД от SAP, ориентированная на быстрые аналитические и транзакционные нагрузки. Ключевая идея — хранение данных преимущественно в оперативной памяти и оптимизация выполнения запросов за счёт колоночного хранения, с сохранением устойчивости (persistency) на диске.

- **In-memory архитектура**: рабочие данные находятся в памяти для минимизации задержек доступа; на диск ведётся журналирование и сохранение страниц данных для восстановления после сбоев.
- **Колоночное и строковое хранение**:
  - **Column Store** — основной формат для аналитики (быстрые агрегации, компрессия, чтение нужных колонок).
  - **Row Store** — используется для определённых сценариев, ближе к классическим OLTP-таблицам.
- **SQL и объекты БД**: поддерживается SQL (диалект SAP HANA) и стандартные объекты — схемы (schema), таблицы, представления (views), индексы/ограничения, процедуры/функции, пользователи и роли.
- **Схемы (Schemas)**: логическое пространство имён для объектов (например, таблица `USERS` может находиться в схеме `MY_SCHEMA` и быть доступной как `MY_SCHEMA.USERS`).
- **Безопасность и доступ**: управление правами через пользователей/роли, привилегии на схемы/таблицы/процедуры; возможна интеграция с корпоративной аутентификацией.
- **HDI (HANA Deployment Infrastructure)**: подход к развертыванию артефактов БД (таблицы/представления и т.п.) через контейнеры; часто применяется в SAP BTP/XS Advanced сценариях.
- **Подключение приложений**: распространённые варианты — **ODBC/JDBC**, а также нативные/HTTP-драйверы в зависимости от стека. В Rust на Windows/Unix часто используют ODBC-драйвер через крейт `odbc-api`.

### Что обычно важно указать в документации/конфигурации подключения

- **Host/Port** (или DSN): адрес и порт сервиса HANA.
- **User/Password** (или иной способ аутентификации).
- **Schema** (опционально): целевая схема для SQL-операций.
- **TLS/SSL параметры** (если включено шифрование).
- **Параметры драйвера**: таймауты, кодировки, режимы курсоров и т.п.

## SAP HANA — типы данных

Ниже — наиболее употребимые **типы данных SAP HANA** (в терминах SQL). Конкретный набор доступных типов может зависеть от версии/конфигурации, но базовые категории общие.

### Числовые

- **TINYINT / SMALLINT / INTEGER / BIGINT**: целые числа разного диапазона.
- **DECIMAL(p, s)**: точная десятичная арифметика (деньги, количества с фиксированной точностью).
- **REAL / DOUBLE**: числа с плавающей точкой (для измерений, где допустима погрешность).

### Строковые (Unicode)

- **NVARCHAR(n)**: основная строка Unicode фиксированного максимального размера (часто используют для имён, кодов, небольших текстов).
- **NCLOB**: большие текстовые данные (длинные описания, документы).

### Дата и время

- **DATE**: дата без времени.
- **TIME**: время без даты.
- **SECONDDATE**: дата-время с точностью до секунд.
- **TIMESTAMP**: дата-время с более высокой точностью (обычно для событий/логов).

### Бинарные и LOB

- **VARBINARY(n)**: бинарные данные ограниченного размера.
- **BLOB**: большие бинарные данные (файлы/изображения/сериализованные структуры).

### Пространственные (GIS), если включены

- **ST_GEOMETRY** (и связанные типы): геометрические данные (точки, полигоны и т.п.) для spatial-запросов.

### Практические рекомендации выбора

- Для идентификаторов/счётчиков: **BIGINT** (или **INTEGER**, если диапазона достаточно).
- Для человекочитаемых строк до 255–500 символов: **NVARCHAR(n)**.
- Для временных меток событий: **TIMESTAMP** (или **SECONDDATE**, если достаточно секунд).
- Для денег/точных величин: **DECIMAL(p, s)** вместо `DOUBLE`.

## SAP HANA — основные группы команд (SQL) с примерами

Ниже — «каркас» команд, которые чаще всего используются при работе с SAP HANA из приложений.

### DDL (Data Definition Language) — создание/изменение объектов

Создание схем/таблиц/представлений, изменения структуры.

```sql
CREATE SCHEMA MY_SCHEMA;

CREATE TABLE MY_SCHEMA.USERS (
  ID   BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  NAME NVARCHAR(255) NOT NULL
);

ALTER TABLE MY_SCHEMA.USERS ADD (EMAIL NVARCHAR(255));
```

### DML (Data Manipulation Language) — изменение данных

Вставка/обновление/удаление строк.

```sql
INSERT INTO MY_SCHEMA.USERS (NAME) VALUES ('User1'), ('User2');

UPDATE MY_SCHEMA.USERS
  SET NAME = 'User1-renamed'
  WHERE ID = 1;

UPSERT MY_SCHEMA.USERS (ID, NAME)
  VALUES (1, 'User1-upserted')
  WITH PRIMARY KEY;

DELETE FROM MY_SCHEMA.USERS
  WHERE ID = 2;
```

### DQL (Data Query Language) — запросы данных (SELECT)

Чтение данных и аналитические запросы.

```sql
SELECT ID, NAME
  FROM MY_SCHEMA.USERS
  ORDER BY ID;

SELECT COUNT(*) AS CNT
  FROM MY_SCHEMA.USERS;
```

### TCL (Transaction Control Language) — управление транзакциями

Фиксация/откат изменений (актуально при выключенном autocommit).

```sql
COMMIT;
ROLLBACK;
```

### DCL (Data Control Language) — права доступа

Выдача/отзыв привилегий. В продуктивных окружениях обычно выполняется администраторами.

```sql
GRANT SELECT, INSERT ON MY_SCHEMA.USERS TO APP_USER;
REVOKE INSERT ON MY_SCHEMA.USERS FROM APP_USER;
```

### Служебные запросы к каталогу (метаданные)

Проверка существования объектов, просмотр системной информации. Часто используют системные представления `SYS.*`.

```sql
-- Проверить, существует ли таблица (пример через SYS.TABLES)
SELECT 1
  FROM SYS.TABLES
  WHERE SCHEMA_NAME = 'MY_SCHEMA'
    AND TABLE_NAME  = 'USERS';

-- Посмотреть колонки таблицы
SELECT COLUMN_NAME, DATA_TYPE_NAME, LENGTH, SCALE
  FROM SYS.TABLE_COLUMNS
  WHERE SCHEMA_NAME = 'MY_SCHEMA'
    AND TABLE_NAME  = 'USERS'
  ORDER BY POSITION;
```

## SAP HANA — триггеры (типы) с примерами

Триггер в SAP HANA — объект БД, который автоматически выполняет код при наступлении события (обычно `INSERT`, `UPDATE`, `DELETE`) для таблицы/представления.

### По времени срабатывания: BEFORE / AFTER

- **BEFORE**: выполняется до изменения данных (часто для валидации или заполнения полей).
- **AFTER**: выполняется после изменения данных (часто для аудита/журналирования).

```sql
-- BEFORE ROW: запретить пустое имя (пример валидации)
CREATE TRIGGER MY_SCHEMA.TR_USERS_BI_VALIDATE
  BEFORE INSERT ON MY_SCHEMA.USERS
  REFERENCING NEW ROW AS N
  FOR EACH ROW
BEGIN
  IF :N.NAME IS NULL OR LENGTH(:N.NAME) = 0 THEN
    SIGNAL SQL_ERROR_CODE 10001 SET MESSAGE_TEXT = 'NAME must not be empty';
  END IF;
END;
```

```sql
-- AFTER ROW: аудит изменения имени (пример)
CREATE TABLE MY_SCHEMA.USERS_AUDIT (
  ID        BIGINT,
  OLD_NAME  NVARCHAR(255),
  NEW_NAME  NVARCHAR(255),
  CHANGED_AT TIMESTAMP
);

CREATE TRIGGER MY_SCHEMA.TR_USERS_AU_AUDIT
  AFTER UPDATE ON MY_SCHEMA.USERS
  REFERENCING OLD ROW AS O NEW ROW AS N
  FOR EACH ROW
BEGIN
  INSERT INTO MY_SCHEMA.USERS_AUDIT (ID, OLD_NAME, NEW_NAME, CHANGED_AT)
  VALUES (:N.ID, :O.NAME, :N.NAME, CURRENT_TIMESTAMP);
END;
```

### По уровню: FOR EACH ROW / FOR EACH STATEMENT

- **FOR EACH ROW**: триггер срабатывает на каждую строку (есть доступ к `OLD ROW`/`NEW ROW`).
- **FOR EACH STATEMENT**: триггер срабатывает один раз на SQL-оператор (удобно для «обвязки», но без построчных данных).

```sql
-- STATEMENT-level: логировать факт массового удаления
CREATE TABLE MY_SCHEMA.OP_LOG (
  OP        NVARCHAR(50),
  HAPPENED_AT TIMESTAMP
);

CREATE TRIGGER MY_SCHEMA.TR_USERS_AD_LOG
  AFTER DELETE ON MY_SCHEMA.USERS
  FOR EACH STATEMENT
BEGIN
  INSERT INTO MY_SCHEMA.OP_LOG (OP, HAPPENED_AT)
  VALUES ('DELETE USERS', CURRENT_TIMESTAMP);
END;
```

### По событию: INSERT / UPDATE / DELETE (+ UPDATE OF)

Триггер можно привязать к одному событию (`INSERT`) или к нескольким. Для `UPDATE` часто ограничивают срабатывание изменением конкретных колонок.

```sql
-- Срабатывать только если обновляли колонку NAME
CREATE TRIGGER MY_SCHEMA.TR_USERS_BU_NAME_ONLY
  BEFORE UPDATE OF NAME ON MY_SCHEMA.USERS
  REFERENCING OLD ROW AS O NEW ROW AS N
  FOR EACH ROW
BEGIN
  -- пример: нормализовать пробелы
  N.NAME := TRIM(:N.NAME);
END;
```

### INSTEAD OF триггеры (для VIEW)

**INSTEAD OF**-триггер применяется к представлениям (VIEW), чтобы переопределить, как обрабатывать `INSERT/UPDATE/DELETE` по view (например, разложить операцию на базовые таблицы).

```sql
-- Пример: view над USERS и INSTEAD OF INSERT
CREATE VIEW MY_SCHEMA.V_USERS AS
  SELECT ID, NAME FROM MY_SCHEMA.USERS;

CREATE TRIGGER MY_SCHEMA.TR_V_USERS_IOI
  INSTEAD OF INSERT ON MY_SCHEMA.V_USERS
  REFERENCING NEW ROW AS N
  FOR EACH ROW
BEGIN
  INSERT INTO MY_SCHEMA.USERS (ID, NAME)
  VALUES (:N.ID, :N.NAME);
END;
```

## SAP HANA — VIEW (представления) с примерами

**VIEW (представление)** — виртуальная таблица, определённая SQL-запросом. Представления используют для:

- упрощения доступа к данным (скрыть сложные `JOIN`/агрегации);
- разграничения доступа (давать права на view вместо «сырых» таблиц);
- стабилизации интерфейса данных для приложений.

### Создание и удаление VIEW

```sql
CREATE VIEW MY_SCHEMA.V_USERS AS
  SELECT ID, NAME
    FROM MY_SCHEMA.USERS;

DROP VIEW MY_SCHEMA.V_USERS;
```

### Обновление определения (замена)

```sql
CREATE OR REPLACE VIEW MY_SCHEMA.V_USERS AS
  SELECT ID, TRIM(NAME) AS NAME
    FROM MY_SCHEMA.USERS;
```

### VIEW с фильтрацией

```sql
CREATE VIEW MY_SCHEMA.V_USERS_ACTIVE AS
  SELECT ID, NAME
    FROM MY_SCHEMA.USERS
    WHERE NAME IS NOT NULL;
```

### VIEW с агрегацией

```sql
CREATE VIEW MY_SCHEMA.V_USERS_STATS AS
  SELECT COUNT(*) AS USERS_COUNT
    FROM MY_SCHEMA.USERS;
```

### VIEW с JOIN (пример «витрины»)

```sql
CREATE TABLE MY_SCHEMA.USER_ROLES (
  USER_ID BIGINT NOT NULL,
  ROLE    NVARCHAR(50) NOT NULL
);

CREATE VIEW MY_SCHEMA.V_USERS_WITH_ROLES AS
  SELECT U.ID,
         U.NAME,
         R.ROLE
    FROM MY_SCHEMA.USERS U
    LEFT JOIN MY_SCHEMA.USER_ROLES R
      ON R.USER_ID = U.ID;
```

## SAP HANA — LIMIT / TOP / OFFSET (ограничение выборки) с примерами

Ограничение количества строк в результатах обычно используют вместе с `ORDER BY`, чтобы результат был детерминированным.

### LIMIT

```sql
SELECT ID, NAME
  FROM MY_SCHEMA.USERS
  ORDER BY ID
  LIMIT 10;
```

### LIMIT + OFFSET (постраничная выборка)

```sql
-- страница 2 при размере страницы 10
SELECT ID, NAME
  FROM MY_SCHEMA.USERS
  ORDER BY ID
  LIMIT 10 OFFSET 10;
```

### OFFSET ... FETCH (SQL-стандартный вариант)

```sql
SELECT ID, NAME
  FROM MY_SCHEMA.USERS
  ORDER BY ID
  OFFSET 10 ROWS FETCH NEXT 10 ROWS ONLY;
```

### TOP (альтернативная форма)

```sql
SELECT TOP 10 ID, NAME
  FROM MY_SCHEMA.USERS
  ORDER BY ID;
```

## SAP HANA — администрирование (примеры)

Ниже — типовые административные операции, которые часто нужны при настройке среды для приложения. В реальных окружениях такие команды обычно выполняются отдельным админ-пользователем.

### Пользователи и пароли

```sql
-- Создать пользователя приложения
CREATE USER APP_USER PASSWORD "StrongPassword#2026";

-- Сменить пароль
ALTER USER APP_USER PASSWORD "NewStrongPassword#2026";

-- Заблокировать / разблокировать пользователя
ALTER USER APP_USER DEACTIVATE USER NOW;
ALTER USER APP_USER ACTIVATE USER NOW;
```

### Роли и привилегии

```sql
-- Создать роль и выдать ей права на схему/таблицу
CREATE ROLE APP_ROLE;
GRANT SELECT, INSERT, UPDATE, DELETE ON MY_SCHEMA.USERS TO APP_ROLE;

-- Назначить роль пользователю
GRANT APP_ROLE TO APP_USER;

-- Отозвать роль/права
REVOKE APP_ROLE FROM APP_USER;
REVOKE DELETE ON MY_SCHEMA.USERS FROM APP_ROLE;
```

### Схемы и права на схему

```sql
CREATE SCHEMA MY_SCHEMA;
GRANT SELECT, INSERT, UPDATE, DELETE ON SCHEMA MY_SCHEMA TO APP_ROLE;
```

### Мониторинг (сессии/соединения) через SYS.*

```sql
-- Активные соединения (набор колонок зависит от версии)
SELECT *
  FROM SYS.M_CONNECTIONS;

-- Активные транзакции
SELECT *
  FROM SYS.M_TRANSACTIONS;

-- Блокировки
SELECT *
  FROM SYS.M_BLOCKED_TRANSACTIONS;
```

## SAP HANA — export / import (выгрузка и загрузка)

В SAP HANA под «export/import» обычно понимают:

- выгрузку/загрузку **данных таблиц** (например, в CSV) для обмена/миграций;
- экспорт/импорт **объектов БД** (схемы/таблицы/представления) как часть переноса;
- массовую загрузку из файлов через ETL/инструменты (в т.ч. через внешние средства).

Ниже — короткие примеры SQL-команд. Синтаксис и доступность опций могут отличаться в зависимости от версии и прав.

### EXPORT (пример выгрузки таблицы)

```sql
-- Выгрузить таблицу в CSV-файлы в указанный каталог на сервере
EXPORT MY_SCHEMA.USERS AS CSV INTO '/hana/shared/exports/users';
```

### IMPORT (пример загрузки таблицы)

```sql
-- Загрузить таблицу из ранее экспортированного набора файлов
IMPORT MY_SCHEMA.USERS AS CSV FROM '/hana/shared/exports/users';
```

### Экспорт результата SELECT (подход через таблицу-витрину)

Иногда удобнее сначала сформировать «срез» данных запросом, записать в отдельную таблицу, а затем экспортировать её.

```sql
CREATE TABLE MY_SCHEMA.USERS_EXPORT AS
  SELECT ID, NAME
    FROM MY_SCHEMA.USERS
    ORDER BY ID;

EXPORT MY_SCHEMA.USERS_EXPORT AS CSV INTO '/hana/shared/exports/users_export';
```

### Практические замечания

- Путь в `EXPORT/IMPORT` обычно относится к **файловой системе сервера**, где запущена SAP HANA, и требует соответствующих прав/доступа.
- В продуктивных системах экспорт/импорт часто регламентирован и выполняется администраторами, либо через управляемые каналы (backup/restore, транспорт, ETL).
